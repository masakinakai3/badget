# ソフトウェア仕様書

## 1. はじめに
### 1.1 目的
本書は、「毎月の予算管理ソフト」の開発に向けた具体的なソフトウェア仕様を定義するものです。`spec.md` に記載された要件を満たすためのアーキテクチャ、機能・非機能の詳細、およびデータ構造を設計のベースとして明確にします。

### 1.2 システム概要
プロジェクトごと、および半期ごとの予算計画と支出実績を管理・可視化するアプリケーションです。予実管理を容易にし、ダッシュボードやレポートによって現在の予算消化状況を直感的に把握できることを目的としています。

---

## 2. アーキテクチャ構成案
ネットワークに依存せず、ユーザーのPC上で完結して動作する「ローカルWebアプリ」として設計します。ユーザーはブラウザからローカルサーバー（localhost）にアクセスして利用する形態となります。

- **フロントエンド:** React (Next.js または Vite) + TypeScript
  - ユーザーのローカルブラウザ上で動作し、使いやすく直感的なSPA（Single Page Application）インターフェースを提供します。
- **バックエンド:** Node.js (Express 等の軽量フレームワーク)
  - ユーザーのPC上でローカルサーバーとして稼働し、フロントエンドからのAPIリクエストを処理します。
- **データベース:** SQLite
  - 軽量でファイルベースのデータベースを採用し、ローカルPC上でのデータ管理（およびファイルコピーでのバックアップ/リストア）を容易にします。
- **スタイリング:** Tailwind CSS などのUIライブラリを利用し、見やすいダッシュボードを構築します。

## 3. 機能仕様

### 3.1 プロジェクト管理機能
- **機能概要:** 管理対象となるプロジェクトを作成・管理する
- **詳細要件:**
  - プロジェクトの新規作成、編集、削除ができる
  - プロジェクトごとに半期の予算総額を設定できる
  - 対象となる「半期」（例: 2024年度 上期 4月〜9月）を設定できる

### 3.2 予算消化項目（カテゴリ）設定機能
- **機能概要:** プロジェクトごとに必要な支出項目をカスタマイズする
- **詳細要件:**
  - 項目（外注費、機材費、ライセンス費、交通費など）の追加・編集・削除
  - プロジェクトごとに独自の項目セットを持てるようにする

### 3.3 予算計画および支出の入力機能
- **機能概要:** 月ごと・項目ごとの予定額と実績額を入力・管理する
- **詳細要件:**
  - **予算計画:** 半期の各月における、項目ごとの「予定額」を入力する機能
  - **支出入力:** 月々の実際の「支出額（実績）」を入力する機能
  - 予定額と実績額から、プロジェクト全体の「予算残高」および各項目の「残高」を自動計算する

### 3.4 ダッシュボード機能
- **機能概要:** 現在の予算状況を一目で把握するためのポータル画面
- **詳細要件:**
  - プロジェクト一覧とそれぞれの予算消化率（％）をプログレスバー等で表示
  - 予算超過、または超過の恐れがある項目に対する警告（アラート）表示
  - 当月の予定額に対する現在の支出状況のサマリー

### 3.5 グラフ・レポート機能
- **機能概要:** 予実データの視覚化
- **詳細要件:**
  - **予実推移グラフ:** 半期間の月別「予算計画」と「実績」の比較（棒グラフ・折れ線グラフ）
  - **支出内訳グラフ:** 項目別の支出割合（円グラフ）

### 3.6 データエクスポート機能
- **機能概要:** 外部システムや報告用へのデータ出力
- **詳細要件:**
  - **CSV出力:** 各プロジェクトの予実データ（計画・実績リスト）のエクスポート
  - **PDF出力:** ダッシュボードやレポートの画面状態を帳票形式で出力

### 3.7 バックアップ・リストア機能
- **機能概要:** データの安全な保管と復旧
- **詳細要件:**
  - すべてのデータを1つのファイル（JSONまたはSQLiteのダンプなど）としてエクスポートするバックアップ機能
  - バックアップファイルからデータを復元するリストア機能

## 4. 非機能仕様

### 4.1 パフォーマンス
- 軽量のフレームワークとポータブルなDBを利用し、インストールから起動、画面遷移までがシームレスに行われること。
- データ保存や集計グラフの表示にかかるレスポンスタイムを1秒以内とする。

### 4.2 ユーザビリティ
- マニュアルを熟読せずとも使える直感的なインターフェース要素（ツールチップ、入力補助、プレースホルダー）を提供する。

### 4.3 メンテナンス性
- **コード品質:** コンポーネント指向を用いたUI設計を行い、再利用性と可読性を高める。
- TypeScriptを活用し、静的型付けによってバグの発生を未然に防ぐ。
- Linter/Formatter (ESLint, Prettier) を導入し、静的解析をパスするコードベースを維持する。

### 4.4 テスト性
- **単体テスト (Unit Test):** 予算の集計ロジックや残高計算など、コアなビジネスロジックに対してテスト（Jest等）を実装する。
- **統合テスト (Integration Test):** データの保存〜グラフ表示など、コンポーネント間の連携に対するテストを行う。

### 4.5 ドキュメント
- **ユーザーマニュアル:** アプリケーション内に簡単なヘルプ/ガイド機能（Markdownベースまたは静的ページ）を組み込む。
- **API/内部ドキュメント:** 開発者向けにソースコード内のコメントダウン（JSDoc等）や、READMEに必要な起動方法を記載する。

## 5. 概念データモデル (Core Entities)

以下は実装時に想定される主要なデータモデルの構造です。

1. **Project (プロジェクト)**
   - `id`: プロジェクトID
   - `name`: プロジェクト名
   - `term_start`: 半期開始月
   - `term_end`: 半期終了月
   - `total_budget`: 半期全体予算総額

2. **Category (予算消化項目)**
   - `id`: カテゴリID
   - `project_id`: 紐づくプロジェクトID
   - `name`: 項目名（外注費など）

3. **BudgetPlan (予算計画)**
   - `id`: 計画ID
   - `project_id`: プロジェクトID
   - `category_id`: カテゴリID
   - `year_month`: 対象年月
   - `planned_amount`: 予定金額

4. **Expense (支出実績)**
   - `id`: 支出ID
   - `project_id`: プロジェクトID
   - `category_id`: カテゴリID
   - `year_month`: 対象年月
   - `date`: 支払日
   - `actual_amount`: 実際の支出金額
   - `note`: 備考/メモ
